중요. 진짜 중요. 제일 중요. 어셈블리 주요 사항! 이거 해결 못하면 xml 백날 바꿔도 모드 터진다!

거인 모더는 JecsTools. 젝스툴즈의 CompOversizedWeapon.dll을 이용해서 거인모드 대형무기 코드를 짠 게 틀림없음.
문제는 좀 많이 낮은 버전을 쓴 것 같다는 점. 왜 그랬는진 모르겠음. 개발 시작했을 때 젝스툴 버전이 그것밖에 없었던가 당시 있던 가이드가 해당 버전에 맞는 것 뿐이었던가. 알 수 없는 일이지만 지금 중요한 건 그게 아니고 최신 버전과의 차이점이 꽤 크다는 거 + 네임스페이스가 겹쳐서 젝스툴 같이 쓰면 터질 거란 거!

아예 대부분 빼버리고 종속성으로 만들어서 젝스툴을 필수 선행 모드로 넣던가! 아니면 네임스페이스 등 명칭을 전부 바꿔서 독립적으로 만들던가!

어쩌면... Enable Oversized Weapons 모드 구성을 이용할 수 있을지도 모른다. 어셈블리를 제거하고 xml을 편집한 뒤 해당 모드를 선행으로 넣거나.
종속성 늘어나는 게 맘에 안 든다면, 해당 C# 코드를 뜯어와서 이름을 바꾸고 이용한다거나.



EOW 모드의 코드를 쓰되 종속성 선행모드가 아니라 해당 코드를 재활용하는 것이므로 네임스페이스를 다 바꿔 충돌 위험을 없앤다.
그리고 C# 코드 작업을 살짝 더 진행한다. 날아가는 거인 무기 투사체의 크기를 들고 있을 때와 동일하게 하는 방식같은 것.
내 기억 상 거인이 무기를 쓸 때 투창같은 건 조준하고 당기는 애니메이션도 있던 것 같은데, 그게 있었든 없었든 이번에는 과감하게 제외.
이 경우 투사체(예: 투창)가 비행 중에도 drawSize와 drawOffset 값을 준수하도록 하기만 하면 ok. 이거 하려면 Projectile.Draw 메서드의 렌더링 로직을 Harmony 패치로 수정해야 할 수도 있음. C# 코딩으로 손댈 부분은 거의 없을 것으로 추정됨.

XML 작업에서는 각 초대형 무기의 graphicData을 해야 하는데 이게 꽤 빡세고 복잡할 가능성이 농후함. 그나마 drawSize를 바꾸면 휴대시와 투척시 다 바뀐다고 함. ChatGPT Rimworld Mod Mentie가 한 말인데 C# 코딩을 그렇게 했을 경우를 상정한 건지 그냥 된단 건지는 모르겠다. 그냥 바닐라에 EOW 모드 넣고 림월드 바닐라의 투창 크기를 EOW에 맞춰서 수치 바꾸는 간단한 모드 하나 넣은 다음 테스트해보면 확실할듯. 그냥 되면 좋겠다~!
